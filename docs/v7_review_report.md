## 2. 全面 Review 发现的重大修正（🚨）与优化（⚠️）

### 🚨 2.1 坡度估算公式存在严重的数学推导错误
* **位置**：`controller_spec.md` 第 913 行（LonController Step 8）
* **现状错误代码**：`road_slope = (accel_mps2 - dvdt / kcoef_gravity) / kcoef_gravity`
* **截图证据**：在截图 `175314_041.png` 中，原 C++ 代码推导为 `slope_ = (acc_fb - dvdt) / 9.8;`
* **物理原理分析**：车辆自带纵向加速度计的读数 $a_{fb}$ 包含了运动加速度 $\dot{v}$ 和重力沿坡道的分量 $g \sin(\theta)$。即：$a_{fb} = \dot{v} + g \sin(\theta)$。因为坡度通常较小，近似 $\sin(\theta) \approx \theta$，由此推导：$\theta = (a_{fb} - \dot{v}) / g$。
* **整改方案**：文档中两次除以了重力系数 `kcoef_gravity`。必须将第 913 行修正为：
  ```python
  road_slope = (accel_mps2 - dvdt) / kcoef_gravity
  ```

### ⚠️ 2.2 IIR 低通滤波器定义混淆
* **位置**：`controller_spec.md` 第 1.3 节与第 3.5 节。
* **审查发现**：
  * 文档 1.3 节定义了一个基于差分方程的 IIR（对应代码：`y = x - alpha * y_prev`，主要用于纵向 `acc_out` 滤波）。
  * 但在 3.5 节（第 558-559 行）电拖横向 `heading_from_dy` 处理时，使用的是标准指数移动平均 (EMA)：`heading_from_dy = raw * alpha + prev * (1 - alpha)`。
* **整改方案**：这两个实现分别对应原代码中的不同逻辑，都没错。但为防止使用者混淆，建议在 1.3 节的基础组件中同时提供这**两类低通滤波器**的清晰定义，并在后续章节调用时注明使用的类型。

## 3. 算法闪光点与优良还原度（💡）

### 💡 3.1 横向偏差（Lateral Error）的符号规范
* **位置**：文档第 305 行及 537 行。
* **实现**：`lateral_error = cos(currt.theta) * dy - sin(currt.theta) * dx`
* **审查评价**：极其精准。完美还原了二维向量叉乘 $\vec{v}_{path} \times \vec{v}_{error}$。准确保证了“车辆在轨迹左侧时误差为正”的 Frenet 符号体系，并且旁边的注释也清清楚楚地标明了符号规则，极大方便了算法的无缝移植。

### 💡 3.2 纵向加速率限制的查表索引选用
* **位置**：文档第 828-829 行。
* **实现**：`acc_up_rate_raw = lookup1d(L3, acc_out_prev)`
* **审查评价**：在大多数控制逻辑中，开发者极易混淆，误用 `speed` 来作为变化率表格的索引。文档准确识别出了此处查表 `L3` / `L4` 的横轴索引是**上一帧的加速度 (`acc_out_prev`)**，这完全符合截图 `175154_539.png` 中索引域为 `[-1.0, 1.0]` 的物理事实，体现了极高的细节捕获能力。

## 5. 新增：算法错误补充（与“逐字复现正确性”分开）

> 说明：本节关注“算法本身是否合理可运行”，不用于否定截图逐字复现结果。

### 5.1 坡度公式数学错误（高优先级）
* **现状**：文档（且与截图一致）为 `road_slope = (accel_mps2 - dvdt / kcoef_gravity) / kcoef_gravity`。
* **问题**：等价于对 `dvdt` 做了双重重力系数归一，量纲与物理意义均不一致。
* **建议修正**：
  ```python
  road_slope = (accel_mps2 - dvdt) / kcoef_gravity
  ```

### 5.2 `history_valid_frames` 变量未定义（高优先级）
* **现状**：Step 8 使用 `if len(history_valid_frames) >= 10:`，但内部状态区未定义该变量。
* **影响**：按该伪代码直接实现会在运行时触发未定义变量错误。
* **建议修正**：统一为已存在缓冲区变量（如 `speed_history`）或在状态区显式定义并维护 `history_valid_frames`。

### 5.3 速度历史单位不一致（中优先级）
* **现状**：注释写 `speed_history` 为 `km/h`，但写入值是 `speed_mps`。
* **影响**：会导致 dvdt 估算量纲混乱，进而影响坡度与扭矩计算精度。
* **建议修正**：二选一并保持一致：
  1) 注释改为 m/s；或  
  2) 写入前将 `speed_mps` 转回 km/h（并同步调整 dvdt 计算公式）。
